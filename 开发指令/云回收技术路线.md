# 云回收小程序 · 技术路线设计

> 配套文档：[云回收项目PRD.md](./云回收项目PRD.md)
> 本文给出符合本项目现状（微信云开发 CloudBase + 单云函数分发模式）的技术实现路线，作为第一版开发依据。

## 1. 总体原则

本项目已有架构是 **微信云开发**，前端在 `miniprogram/`，后端是 `cloudfunctions/` 下的单个云函数 `quickstartFunctions`，前端用 `data.type` 指定动作、云函数内 `switch(event.type)` 分发。

技术路线遵循三条原则，避免推倒重来：

1. **沿用单云函数 + type 分发**：所有回收业务后端动作继续加在 `quickstartFunctions` 的 `switch` 里，不新建云函数。
2. **沿用云数据库 + 云存储**：订单、地址、品类存数据库；用户拍的照片存云存储，数据库里只存 `fileID`。
3. **复用现有组件与跳转方式**：弹窗复用 `components/cloudTipModal`，页面跳转继续用 `wx.navigateTo`，错误处理沿用对 `errMsg` 的匹配。

> 第一版聚焦 PRD 第 9 节范围：**用户提交订单 + 查看订单进度**。运营/回收员后台先用「微信云开发控制台 + 简单管理页」过渡，不做独立 Web 后台。

## 2. 技术选型

| 层 | 技术 | 说明 |
|---|---|---|
| 前端框架 | 微信小程序原生（WXML/WXSS/JS） | 与现有代码一致，不引入 uni-app/Taro |
| 后端 | 云函数 `quickstartFunctions` | 单函数 + type 分发 |
| 数据库 | 云开发文档型数据库 | 集合见第 4 节 |
| 文件存储 | 云存储 | 存物品照片，库里存 fileID |
| 登录态 | `cloud.getWXContext().OPENID` | 云函数侧拿 openid，无需自建登录 |
| 图片展示 | `wx.cloud.getTempFileURL` | fileID 换临时链接展示 |
| 定位/地图 | `wx.getLocation` + `wx.chooseLocation` + 腾讯位置服务逆地理编码 | 首页当前地区、地址页地图选点（见第 9 节） |

在 `app.json` 的 `pages` 中新增以下页面（保留现有 index/example 用于调试，第一版上线前可删）：

|---|---|---|

| `pages/category/index` | 可回收品类列表（含「不确定？拍照提交」入口） | 用户目标 |
| `pages/order-create/index` | 提交订单：选品类/填重量/拍照/选地址/选时间 | 4.1 / 4.2 |
| `pages/address/list` | 地址列表（选择 + 管理） | 地址管理 |

## 4. 数据库设计（云开发集合）

新增 4 个集合。命名小写、字段用驼峰。涉及用户归属的集合带 `_openid`，**由云函数手动写入**（前端走云函数，云函数是管理员身份，不会自动带 openid）。

### 4.1 `categories` 可回收品类
平台维护，用户只读。
```
{ _id, name: "纸箱", unit: "kg", priceRef: "0.8元/kg", icon: fileID, sortOrder: 1, enabled: true }
```

### 4.2 `addresses` 用户地址
```
{ _id, _openid, contactName, phone, region, detail, isDefault: false, createTime }
```

### 4.3 `orders` 回收订单（核心）
```
{
  _id, _openid,
  orderNo,                       // 业务单号，云函数生成
  source: "category" | "photo",  // 按品类 / 拍照提交
  items: [ { categoryId, categoryName, estWeight, estCount } ],
  photos: [ fileID ],            // 拍照提交的图片
  addressSnapshot: { contactName, phone, region, detail }, // 下单时地址快照，防止后续地址被改
  appointDate, appointSlot,      // 预约上门：日期 + 时间段
  remark,
  status,                        // 见 4.4 状态机
  estimatePrice,                 // 平台估价
  finalWeight, finalPrice,       // 回收员反馈的实际重量/金额
  recyclerId,                    // 分配的回收员
  createTime, updateTime
}
```

### 4.4 订单状态机 `status`
对应 PRD 业务流程，前端按状态渲染进度条：
```
submitted     已提交（待平台确认）
confirmed     平台已确认/已估价
assigned      已分配回收员
recycling     上门回收中
completed     已完成（含 finalPrice）
canceled      已取消
rejected      拍照物品不可回收
```
状态只允许后端推进，前端只读展示。

### 4.5 数据库权限（关键）
本项目**前端不直连数据库，全部走云函数**，因此安全边界在云函数而非数据库权限。在云开发控制台设置：
- `categories`：所有人可读，仅管理端可写（品类是公开数据）。
- `addresses` / `orders`：**所有用户不可读写**（仅云函数/管理端可访问）。

> ⚠️ 不要设成"仅创建者可读写"——那是给"前端直连数据库"用的模型。我们的数据隔离由**云函数内 `where({ _openid })`** 实现，且 `_openid` 必须在云函数里手动写入（见第 6 节）。

## 5. 云函数动作（quickstartFunctions/index.js 新增 case）

在现有 `switch(event.type)` 中按以下 type 扩展。建议把回收相关逻辑拆到子文件再 require 进来（index.js 已有注释预留这种写法），避免单文件过长。

| type | 入参 | 作用 |
|---|---|---|
| `initRecycleDB` | - | 一键创建上述集合并写入示例品类（仿照现有 `createCollection`，已存在则 catch 返回 success） |
| `listCategories` | - | 返回启用的品类列表 |
| `getAddressList` | - | 按 `_openid` 返回地址 |
| `saveAddress` | address | 新增/更新地址，处理默认地址互斥 |
| `deleteAddress` | id | 删除地址 |
| `createOrder` | 订单数据 | 生成 orderNo、写地址快照、手动写 _openid、拼 summary、status=submitted |
| `getOrderList` | statusGroup?, page?, pageSize? | 按 openid + 状态组分页返回，返回 `{list,total,hasMore}` |
| `getOrderDetail` | id | 返回订单详情，并把 photos 的 fileID 换成临时 URL |
| `cancelOrder` | id | 用户取消（仅 submitted/confirmed 可取消） |
| `getTempFileURL` | fileIDs | 批量换图片临时链接（也可前端直接 `wx.cloud.getTempFileURL`） |

> 运营/回收员操作（确认、估价、改状态、填实际金额）第一版**先在云开发控制台手动改库**，或加 `adminUpdateOrderStatus` 一个 case 配合一个隐藏管理页。等业务跑通再做独立后台。

## 6. 关键技术点

### 6.0 用户身份与数据隔离（务必先看）
前端走云函数，云函数是**管理员身份**，写库时**不会自动写入 `_openid`**。因此：
```js
// 写入：手动塞 _openid
const { OPENID } = cloud.getWXContext();
await db.collection("orders").add({ data: { ...order, _openid: OPENID } });

// 查询：用 _openid 过滤，保证只拿到自己的
await db.collection("orders").where({ _openid: OPENID }).get();
```
- 所有"我的数据"接口（地址、订单）都必须这样写 + 查。
- 这是安全隔离的唯一手段（数据库权限已收紧为前端不可访问，见 4.5）。

### 6.1 图片上传（拍照提交）
1. 前端 `wx.chooseMedia` 选图（最多 9 张）→ `wx.cloud.uploadFile` 上传到云存储，路径 `recycle/{openid}/{时间戳}.jpg`。
2. 上传返回的 `fileID` 存进 `orders.photos`。
3. 展示时用 `wx.cloud.getTempFileURL` 把 fileID 换成可访问 URL。
4. **图片孤儿（R12，已知权衡）**：先传后提交，未提交的图片会成为孤儿文件。第一版接受，后期用定时云函数扫描清理无订单引用的文件；前端删图时可顺手 `wx.cloud.deleteFile`。

### 6.2 地址快照
下单时把当前地址完整复制到 `orders.addressSnapshot`，不要只存 addressId——避免用户日后改了/删了地址导致历史订单地址错乱。

### 6.3 单号生成
`createOrder` 在云函数内生成 `orderNo`，如 `日期 + 随机/自增`，仅作展示用，主键仍用 `_id`。

### 6.4 错误处理
沿用现有套路：云函数失败返回 `{ success:false, errMsg }`，前端匹配 `Environment not found` / `FunctionName ... could not be found` 时用 `cloudTipModal` 提示「环境未配置 / 云函数未上传」。

## 7. 开发实施顺序（建议）

1. **环境就绪**：在 `app.js` 填入云开发环境 ID；部署 `quickstartFunctions`。
2. **建库**：实现并调用 `initRecycleDB`，在控制台核对集合与权限。
3. **品类与首页**：`listCategories` + `pages/home`、`pages/category`。
4. **地址模块**：地址相关 4 个 case + 地址两个页面。
5. **下单主流程**：`createOrder` + `pages/order-create`（含拍照上传）。
6. **订单查看**：`getOrderList` / `getOrderDetail` + 列表/详情页与状态进度条。
7. **管理过渡**：控制台手动改状态验证全链路，跑通后再评估管理页/回收员端。

## 8. 暂不在第一版范围

- 在线支付/结算（仅记录金额，不收付款）。
- 独立运营后台与回收员 App/小程序端（先用控制台过渡）。
- 消息推送/订阅消息（状态变更通知）——可作为第二版增强。
- 物品 AI 自动识别（拍照仅人工判断）。

## 9. 定位与地图集成

### 9.1 能力与落点
| 能力 | 页面 | 实现 |
|---|---|---|
| 首页当前地区 | `pages/home/index` | `wx.getLocation` 取经纬度 → 腾讯逆地理编码 → 左上角展示 |
| 地址地图选点 | `pages/address/edit` | `wx.chooseLocation` 选点 → 回填详细地址 + 逆地理编码拆省市区 |

### 9.2 工具封装 `miniprogram/utils/map.js`
统一封装定位与逆地理编码，页面不直接拼接腾讯接口：
- `getLocation()`：`wx.getLocation({ type: 'gcj02' })` 取当前经纬度。
- `reverseGeocode(lat, lng)`：调腾讯位置服务 `https://apis.map.qq.com/ws/geocoder/v1/`，返回 `{ address, formatted, province, city, district, street, ... }`。
- `getCurrentAddress()`：组合上面两步，首页一次拿到当前中文地址。

### 9.3 腾讯位置服务 Key（外部依赖）
- Key 配置在 `utils/map.js` 顶部 `MAP_KEY`。
- 控制台（https://lbs.qq.com/）需为该 key：① 勾选 **WebServiceAPI**；② 把本小程序 **AppID 加入白名单**，否则 `wx.request` 调逆地理编码被拒。
- `wx.chooseLocation` 本身用微信内置地图、**不依赖** key；仅"逆地理编码拆省市区"和"首页当前地区"依赖 key。key 缺失时地图选点仍可用（只回填详细地址），首页降级显示提示。

### 9.4 权限声明（app.json）
```jsonc
"permission": { "scope.userLocation": { "desc": "用于获取当前位置以显示所在地区并便于填写回收地址" } },
"requiredPrivateInfos": ["getLocation", "chooseLocation"]
```

### 9.5 降级策略
- 定位/逆地理失败、用户拒绝授权：首页显示"点击定位"（可点击重试），地址页提示手动选省市区，**均不阻塞主流程**。

## 10. 开发阶段临时关闭登录鉴权

- 现状：`utils/auth.js` 的 `checkLogin()` 已临时改为直接 `return true`，所有页面放行，原始登录校验逻辑保留在其下方（被 `return true` 短路）。
- 目的：开发联调期免登录直接走主流程；**上线前必须恢复**——删除 `checkLogin` 顶部的 `return true` 即可启用原逻辑。
- 注意：鉴权关闭仅影响前端页面进入；后端数据隔离仍依赖云函数内 `OPENID` + `where({_openid})`，不受影响。

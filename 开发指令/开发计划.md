## 0. 计划总览

### 0.1 技术基线（不变更）
- **前端**：微信小程序原生（WXML/WXSS/JS），目录 `miniprogram/`。
- **后端**：单云函数 `quickstartFunctions` + `switch(event.type)` 分发，目录 `cloudfunctions/quickstartFunctions/`。
- **数据库**：云开发文档数据库，4 个集合 `categories / addresses / orders`（+ 后期 `admins`）。
- **存储**：云存储，库里只存 `fileID`。
- **身份**：云函数内 `cloud.getWXContext().OPENID`，**写入手动塞 `_openid`，查询用 `where({_openid})`**。

### 0.2 第一版范围（MVP）
用户端：品类浏览、按品类预约、拍照提交、地址管理、订单列表与详情、上门前取消。
后端：上述能力对应的云函数 type。
运营/回收员：第一版走**云开发控制台**手动改库，不做独立后台。

### 0.3 开发阶段与里程碑

| 阶段 | 名称 | 产出 | 里程碑（验收标志） |
|---|---|---|---|
| **P0** | 环境与脚手架 | 填环境 ID、部署云函数、建库 | 控制台能看到 3 个集合 + 示例品类 |
| **P1** | 后端云函数全量 | 9 个业务 type 实现并自测 | 开发者工具调试器单独调用每个 type 返回正确 |
| **P2** | 前端框架与首页 | tabBar、首页、品类页 | 能浏览品类列表 |
| **P3** | 地址模块 | 地址列表 + 编辑页 | 能增删改地址、默认地址唯一 |
| **P4** | 下单主流程 | 下单页（含拍照上传） | 两种方式都能成功提交订单 |
| **P5** | 订单查看 | 列表（分页/筛选）+ 详情（进度条/取消） | 全链路：下单→列表→详情→取消 |
| **P6** | 联调与收尾 | 控制台推进状态验证、清理模板代码 | 状态机全流程跑通，删除 sales 示例 |

> 建议顺序就是 P0→P6。后端（P1）先于前端业务页（P3–P5），因为前端每个页面都依赖对应接口。

## 2. 阶段 P1 · 后端云函数

**实现位置**：`cloudfunctions/quickstartFunctions/index.js` 的 `switch(event.type)`。
**建议拆分**：回收业务逻辑量较大，建议拆子文件（如 `recycle/orders.js`、`recycle/address.js`）再 `require` 进 index.js，避免单文件过长。
**统一返回**：`{ success, data?, errMsg? }`，错误码取值见接口契约 0.4。

### 实现顺序与要点

| # | type | 依赖文档 | 关键实现点（务必照此做） |
|---|---|---|---|
| 2.1 | `initRecycleDB` | 接口 1.1 | 创建 3 集合 + 写 5 条示例品类；已存在则 catch 仍返回 success（仿 createCollection） |
| 2.2 | `listCategories` | 接口 2.1 | `where({enabled:true})`，按 `sortOrder` 升序 |
| 2.3 | `getAddressList` | 接口 3.1 | `where({_openid})`，默认地址排最前 |
| 2.4 | `saveAddress` | 接口 3.2 + **R10** | 校验必填→`PARAM_INVALID`；`isDefault=true` 时先把其它地址置 false 再写；**首次新增自动设默认** |
| 2.5 | `deleteAddress` | 接口 3.3 + **R11** | 校验归属→`ADDRESS_NOT_FOUND`；删的是默认地址时，把最近创建的另一个设默认 |
| 2.6 | `createOrder` | 接口 4.1 + **R13/R15** | 见下方「createOrder 实现清单」 |
| 2.7 | `getOrderList` | 接口 4.2 | `where({_openid})` + statusGroup 映射 + 分页；返回 `{list,total,hasMore}`，按 createTime 倒序 |
| 2.8 | `getOrderDetail` | 接口 4.3 | 校验归属→`ORDER_NOT_FOUND`；把 photos 的 fileID 批量换临时 URL 返回 `photoUrls` |
| 2.9 | `cancelOrder` | 接口 4.4 | 仅 `submitted/confirmed` 可取消，否则 `ORDER_STATUS_INVALID`；置 `canceled` + updateTime |
| 2.10 | `getTempFileURL` | 接口 4.5 | 可选；批量 `cloud.getTempFileURL` |

**createOrder 实现清单（最核心）：**
1. 取 `OPENID`；校验入参（source、地址、时间段必填）→ `PARAM_INVALID`。
2. 校验 `addressId` 归属当前 openid → `ADDRESS_NOT_FOUND`；查出后转成 `addressSnapshot`（去掉 isDefault/createTime）。
3. **source=category**：校验最低起收量（默认总重量 ≥ 5kg，Q2）→ `BELOW_MIN_QUANTITY`；source=photo 跳过。
4. 生成 `orderNo`：`yyyyMMdd + 6 位随机`，**日期部分按北京时间（UTC+8）算**（R13 时区坑）。
5. 拼 `summary` 落库（接口 4.1 规则）：category=`"{首品类} 约{首项量}{unit} 等{N}类"`；photo=`"拍照提交 {N}张"`。
6. 写库：`data._openid = OPENID`、`status="submitted"`、`createTime/updateTime=Date.now()`（毫秒时间戳）。
7. 返回 `{ _id, orderNo }`。
8. 幂等（R15）：第一版至少靠前端按钮防抖；如做后端，入参加 `clientToken` 去重。

### P1 验收
用开发者工具「云函数 → 本地/云端调试」逐个调用，每个 type 入参/出参符合接口契约；越权场景（查他人订单）返回对应错误码。

## 5. 阶段 P4 · 下单主流程（最核心页面）

**页面**：`pages/order-create/index`，按 query `source` 渲染不同表单。

| # | 任务 | 要点 |
|---|---|---|
| 5.1 | 页面骨架 | onLoad 读 `source`；进入时 `getAddressList` 取默认地址；公共区块：地址、预约时间（日期 appointDate + 时间段 appointSlot）、备注 |
| 5.2 | source=category 表单 | 渲染所选品类列表，每项填**预估重量/件数**（按 unit）；前端可做最低起收量预校验提示 |
| 5.3 | source=photo 表单 | `wx.chooseMedia` 选图（最多 9 张）→ 逐张 `wx.cloud.uploadFile`（路径 `recycle/{openid}/{ts}.jpg`）→ 收集 fileID；可加文字描述；删图时同步 `wx.cloud.deleteFile`（R12 缓解孤儿） |
| 5.4 | 地址选择 | 点地址区 → address/list（选择模式）→ 回填 addressId |
| 5.5 | 提交 | 校验必填（地址/品类或图片/重量数字）→ 调 `createOrder`；**提交即禁用按钮 loading**（R15 防抖）；成功 `wx.redirectTo` 到 order-detail?id=新id；失败按错误码 toast（如 `BELOW_MIN_QUANTITY`→"未达最低起收量"） |

### P4 验收
两种方式均能成功创建订单；拍照能上传并存 fileID；各校验边界（未选地址/未达起收量/重量非法）提示正确。

## 10. 关键风险与铁律（编码时贴在显眼处）

1. **`_openid` 必须在云函数手动写**，查询必须 `where({_openid})` —— 这是唯一的数据隔离手段。
2. **前端一律不直连数据库**，全部走 `quickstartFunctions`。
3. **地址存快照不存 addressId** —— 防止改/删地址影响历史订单。
4. **时间存毫秒时间戳**；orderNo 日期部分按北京时间算（UTC+8）。
5. **不新建云函数**，所有后端动作加在 `quickstartFunctions` 的 switch。
6. **图片先传后提交**会产生孤儿，第一版接受，前端删图同步删云存储缓解。

# 云回收小程序产品需求文档 PRD

> 配套：[云回收技术路线.md](./云回收技术路线.md) · [云函数接口契约.md](./云函数接口契约.md) · [数据字典.md](./数据字典.md) · [页面与交互流程.md](./页面与交互流程.md)

## 1. 产品概述

云回收是一款面向普通居民、社区用户和小型商户的回收预约小程序。

用户可以通过小程序查看可回收品类、提交回收预约、拍照上传不确定品类的物品，并填写上门地址和预约时间。平台收到订单后，对订单进行确认、估价、联系用户并安排后续回收服务。

**第一版产品重点围绕"提交回收订单"和"查看订单进度"展开。** 运营与回收员的订单处理在第一版通过微信云开发控制台过渡，独立管理后台作为后期阶段单独开发（见第 11 节）。

## 2. 产品目标

### 2.1 用户目标

- 快速了解哪些物品可以回收。
- 已知回收品类时，可以直接预约回收。
- 不知道物品是否可回收时，可以拍照提交。
- 提交订单时可以填写或选择上门地址。
- 进入小程序时能看到当前所在城市/地区，新增地址时可借助地图快速选点。
- 可以查看订单处理状态和回收结果。

### 2.2 平台目标

- 接收用户提交的回收订单。
- 查看用户上传的物品信息、图片、地址和联系方式。
- 判断拍照提交的物品是否可回收。
- 更新订单状态。
- 记录实际回收结果和最终金额。

## 3. 用户角色

第一版**运营与回收员为两个分开的角色**，职责不同。

| 角色 | 说明 | 核心操作 |
|---|---|---|
| 普通用户 | 提交回收需求的用户 | 查看品类、拍照提交、预约回收、管理地址、查看订单、上门前取消订单 |
| 平台运营人员 | 接单与调度的人员 | 查看订单、联系用户、确认订单、人工估价、判断拍照物品是否可回收、分配回收员、更新订单状态 |
| 回收员 | 执行上门回收的人员 | 查看分配给自己的订单、上门回收、反馈实际重量与最终金额、完成订单 |

> 第一版运营和回收员均通过云开发控制台操作数据（不提供独立界面）；后期开发管理后台后，二者将有各自的操作界面。

## 4. 核心业务流程

### 4.1 按品类预约回收流程

```text
用户进入小程序
→ 查看首页
→ 点击我要回收
→ 选择回收品类（可多选）
→ 填写预估重量/数量（需满足最低起收量）
→ 选择或填写回收地址
→ 选择预约时间（日期 + 时间段）
→ 提交订单（状态：已提交）
→ 运营确认订单并人工估价（状态：已确认）
→ 运营分配回收员（状态：已分配）
→ 回收员上门回收（状态：回收中）
→ 回收员反馈实际重量与金额（状态：已完成）
```

### 4.2 拍照提交回收流程

```text
用户进入小程序
→ 点击拍照提交
→ 拍照或从相册选图（最多 9 张）+ 填写物品描述
→ 选择或填写回收地址
→ 选择预约时间（日期 + 时间段）
→ 提交订单（状态：已提交）
→ 运营查看图片并判断是否可回收
   ├─ 可回收 → 人工估价（状态：已确认）→ 后续同 4.1
   └─ 不可回收 → 标记拒收并填写原因（状态：暂不可回收，流程结束）
```

### 4.3 取消订单流程

```text
用户在订单详情点击取消
→ 校验订单状态
   ├─ 已提交 / 已确认 → 允许取消（状态：已取消）
   └─ 已分配及之后 → 不允许，提示联系平台
```

## 5. 功能需求

### 5.1 可回收品类
- 由平台维护品类清单，用户只读。
- 每个品类含：名称、计量单位（kg 或 件）、参考价文案、图标、排序、是否上架。
- **参考价仅作展示**，不参与金额自动计算。
- 第一版示例品类（最终清单待平台确认）：纸箱/废纸、塑料瓶/塑料、易拉罐/金属、旧衣物、旧家电。

### 5.2 提交回收订单
两种提交方式，共用地址、时间、备注：

| | 按品类预约 | 拍照提交 |
|---|---|---|
| 物品信息 | 选择品类（可多选）+ 填预估重量/件数 | 拍照/选图，最多 9 张 + 物品描述 |
| 最低起收量 | 需满足（见 5.3） | 不强制（由运营看图判断） |
| 地址 | 必填，从地址列表选 | 同左 |
| 预约时间 | 必填，日期 + 时间段 | 同左 |
| 备注 | 选填 | 选填 |

### 5.3 最低起收量
- 平台设定最低起收量阈值（如总重量 ≥ 5kg 或总件数 ≥ 1 件，**具体阈值待平台确认**）。
- 按品类预约时，若预估总量低于阈值，提交时提示用户"未达最低起收量"，不允许提交。
- 拍照提交不强制此校验。

### 5.4 预约时间
- 用户选择**日期 + 时间段**（时间段如 9:00-12:00、12:00-18:00，具体分段待确认）。
- 可预约时间范围**不限制**，用户可选未来任意日期。

### 5.5 地址管理
- 用户可新增、编辑、删除地址，可设置默认地址（默认地址唯一）。
- **地图选点**：新增/编辑地址时可调用微信地图选点（`wx.chooseLocation`），选中后自动回填详细地址，并通过逆地理编码自动填充省市区，用户也可手动选择/修改。
- 下单时选择一个地址，系统将该地址**快照**保存到订单（后续修改/删除地址不影响历史订单）。

### 5.5.1 当前位置展示
- 首页左上角展示用户**当前所在地区**（如"上海市浦东新区"），通过定位 + 逆地理编码获得。
- 仅作展示与体验优化，不参与下单地址（下单仍以地址列表中选择的地址为准）。
- 用户拒绝定位授权或定位失败时，降级显示"点击定位"，可点击重试，不阻塞任何功能。

### 5.6 订单查看
- 用户可查看自己的订单列表（支持按状态筛选）和订单详情。
- 详情展示：状态进度、物品信息/图片、地址、预约时间、备注、平台估价、最终重量与金额、拒收原因（如有）。

### 5.7 取消订单
- 用户在「已提交」「已确认」状态下可自助取消；「已分配」及之后状态不可自助取消，需联系平台。

### 5.8 运营与回收员处理（第一版经控制台）
- 运营：确认订单、人工估价、判断拍照物品可否回收（不可回收则拒收并填原因）、分配回收员、推进状态。
- 回收员：查看分配给自己的订单、上门回收、回填实际重量与最终金额、完成订单。

## 6. 订单状态机

| 状态 | 含义 | 谁可推进 | 用户界面文案 |
|---|---|---|---|
| `submitted` | 已提交，待运营确认 | 系统（下单） | 已提交，等待确认 |
| `confirmed` | 运营已确认/已估价 | 运营 | 已确认，等待上门 |
| `assigned` | 已分配回收员 | 运营 | 回收员处理中 |
| `recycling` | 上门回收中 | 回收员 | 回收员处理中 |
| `completed` | 已完成（含实际重量/金额） | 回收员 | 已完成 |
| `canceled` | 用户取消 | 用户 | 已取消 |
| `rejected` | 拍照物品不可回收（含原因） | 运营 | 暂不可回收 |

**流转规则：**
- 用户仅可：`submitted`/`confirmed` → `canceled`。
- `completed` / `canceled` / `rejected` 为终态。
- 其余流转由运营/回收员推进。

## 7. 金额与计价规则

- 品类参考价仅展示，不自动计算订单金额。
- 运营确认时可填写**人工估价**（estimatePrice），供用户参考。
- 回收员完成时回填**实际重量**（finalWeight）与**最终金额**（finalPrice），作为最终结算依据。
- 第一版**不含线上支付**，仅记录金额，线下结算。

## 8. 数据与存储

- 物品照片存云存储，订单仅记录 fileID。
- 订单、地址按用户隔离（仅本人可见）。品类所有人可读。
- 详细字段见 [数据字典.md](./数据字典.md)。

## 9. 第一版范围（MVP）

**包含：**
- 用户端：品类浏览、按品类预约、拍照提交、地址管理、订单列表与详情、上门前取消。
- 后端：云函数完成上述能力（见 [云函数接口契约.md](./云函数接口契约.md)）。
- 运营/回收员处理：经云开发控制台操作。

**不包含（见第 10/11 节）。**

## 10. 暂不在第一版范围

- 线上支付/结算（仅记录金额）。
- 消息推送/订阅消息（状态变更通知）——可作为第二版增强。
- 物品 AI 自动识别（拍照仅人工判断）。
- 优惠券、积分、邀请等运营功能。

## 11. 后期阶段：管理后台（排在最后）

第一版上线、主流程跑通后，再独立开发管理后台，目标是替代云开发控制台的人工操作：

- **运营端**：订单列表与筛选、确认/估价、判断可否回收（拒收填原因）、分配回收员、品类维护。
- **回收员端**：查看分配订单、上门打卡、回填实际重量与金额、完成订单。
- 形态待定（独立小程序端 / Web 后台），与第一版用户端共用同一套云函数与数据库。
